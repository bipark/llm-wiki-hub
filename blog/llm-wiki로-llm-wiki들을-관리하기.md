# LLM-Wiki 로 프로젝트 관리하기 — LLM-Wiki-Hub

요즘 작업하는 프로젝트들을 대부분 LLM-Wiki 라는 방식으로 관리하고 있다. 그러다 보니
어느 순간 "이 위키들을 또 위키처럼 묶어서 보면 어떨까" 하는 생각이 들었고, 그렇게
나온 게 LLM-Wiki-Hub 다. 이 글은 LLM-Wiki 가 뭔지, 그게 왜 좋았는지, 그리고 거기서
한 발 더 나가서 프로젝트들을 한눈에 보는 도구를 어떻게 만들게 됐는지에 대한 기록이다.

거창한 프레임워크 소개라기보다는, 그냥 내가 이렇게 해봤더니 괜찮더라 하는 경험담에
가깝다.

## 프로젝트가 늘어나면서 생긴 문제

최근에 LLM을 기반으로 코딩을 하며 한번에 처리하는 프로젝트 갯수가 늘어 나기 시작했다.
ESP32 펌웨어가 몇 개, RTK GPS, LiDAR 처리, 온도 수집 서버까지. 각자 독립된 git 저장소고, 
서로 합칠 이유도 없다. 문제는 이게 대여섯 개를 넘어가는 순간부터 머릿속에서 상태 추적이 슬슬 안 되기
시작했다는 거다.

"지난주에 이거 어디까지 했더라", "이건 왜 막혀 있었지", "다음에 뭐 하기로 했지"
같은 질문이 떠오를 때마다 해당 디렉토리로 옮겨가서 `git log` 를 더듬고, 코드를 다시
읽고, 기억을 끄집어냈다. 컨텍스트가 전부 내 머릿속에만 있으니까, 한동안 안 본
프로젝트로 돌아갈 때마다 비용이 컸다.

그리고 이건 나만의 문제가 아니었다. Claude Code 같은 LLM 에이전트랑 같이 일하다 보면,
새 세션을 열 때마다 이 친구도 똑같이 맨바닥에서 시작한다. 매번 코드를 처음부터 읽혀야
하고, "왜 이렇게 결정했는지"는 어디에도 안 남아 있으니 같은 설명을 반복하게 된다.

## LLM-Wiki 라는 방식

이 문제를 풀려고 도입한 게 LLM-Wiki 다. 내가 만든 개념은 아니고, Andrej Karpathy 가
[gist 한 장](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 으로
정리해둔 패턴을 가져와서 내 프로젝트들에 맞게 변형해 쓰고 있는 거다.

핵심 아이디어는 이렇다. 보통 RAG 라고 하면 문서를 잔뜩 넣어두고 질문할 때마다 관련
청크를 꺼내 쓴다. 그런데 Karpathy 의 이야기는 그렇게 하면 LLM 이 매 질문마다 지식을
맨바닥에서 다시 찾게 된다는 거였다. 쌓여서 누적되는 히스토리가 없다. 대신 LLM 이 직접 위키를 만들고
유지하게 하면, 지식이 한 번 정리된 채로 계속 쌓여간다.

RAG 가 "그때그때 찾는다"라면 LLM-Wiki 는 "미리 정리해서 쌓아둔다"에 가깝다고 보면
될 것 같다. 둘이 배타적인 건 아니지만 방향이 좀 다르다.

내가 인상 깊었던 건 Karpathy 의 이 표현이었다. 지식 베이스를 유지할 때 지루한 부분은
읽거나 생각하는 게 아니라 부기(bookkeeping) 라는 것. 교차 참조 갱신하고, 중복
정리하고, 오래된 내용 솎아내는 그 잡일 말이다. 그런데 LLM 은 그걸 지루해하지도 않고
빼먹지도 않는다. 그러니까 사람은 큐레이션과 판단을 맡고, 그 귀찮은 정리 작업은 LLM 에게
넘기자는 거다.

실제 구조는 세 겹으로 나눈다.

- **Raw sources** — 소스 코드, 외부 사양서, 데이터시트 같은 1차 자료. 사람이 만들고
  LLM 은 읽기만 한다.
- **Wiki** — `docs/*.md` 에 쌓는 계획·결정 기록·근거·가이드. 이건 LLM 이 자유롭게
  유지·정리한다.
- **Schema** — `CLAUDE.md` 같은 운영 규약. 위키를 어떻게 굴릴지 적어두고, 사람과 LLM 이
  같이 다듬어간다.

여기에 길잡이 역할을 하는 파일 두 개를 둔다. 위키 페이지 목록을 모아둔 `INDEX.md` 와,
결정·변경을 시간 순으로만 쌓는 `LOG.md`. 원칙은 하나, 코드와 위키는 늘 맞아떨어져야
한다는 것. 어긋나면 둘 중 한쪽을 바로 고친다.

이걸 몇 달 해보니 확실히 편했다. 새 세션을 열어도 LLM 이 위키부터 읽고 맥락을 빠르게
복원했고, "왜 이렇게 했는지"가 LOG 와 결정 기록에 남으니 나중에 나도 헷갈리지 않았다.
실제로 한 프로젝트의 로그에는 이 패턴을 도입한 효과를 "새 LLM 세션의 초기 컨텍스트
회복 시간 단축"이라고 적어두기도 했다.

## "이 위키들을 또 위키처럼 묶으면?"

그렇게 프로젝트마다 LLM-Wiki 를 붙이다 보니 위키가 여러 개가 됐다. 그런데 정작 이걸
전체로 가로질러 보는 화면이 없었다. 어느 프로젝트가 막혀 있는지, 어디가 한참 손을 안
댔는지, 통틀어 남은 할 일이 뭔지를 한눈에 보려면 결국 또 디렉토리를 하나씩 돌아야 했다.

그러다 문득, 각 프로젝트가 이미 자기 상태를 평문으로 들고 있다는 데 생각이 미쳤다.
`STATUS.md` 에 지금 상태가 있고, 할 일은 할일 파일에, 결정은 `docs/adr/` 에 있다.
그러면 이걸 그냥 긁어모으기만 해도 전체 현황판이 될것 같았다. 새 도구를 위해
프로젝트마다 뭔가를 더 심을 필요도 없었다. 이미 거기 있는 걸 읽기만 하면 됐으니까.

말하자면 LLM-Wiki 의 발상 — 평문으로 쌓아두고, 단일 출처를 두고, 정리는 자동으로 —
을 프로젝트 하나가 아니라 **여러 프로젝트 위에 한 층 더** 올리는 셈이었다. 그렇게 만든
게 LLM-Wiki-Hub 다.

## LLM-Wiki-Hub 가 하는 일

한 줄로 줄이면, 흩어진 프로젝트들을 건드리지 않고 그 위에 얹어서 한 화면으로 보는
읽기 전용 대시보드다. 내가 실제로 하는 건 `projects.txt` 에 경로를 한 줄씩 적는 것
정도다.

```
~/work/esp32-mesh
~/work/rtk-gps
~/work/lidar-monitor
```

그리고 스크립트를 돌리면,

```
python scripts/aggregate.py
```

`wiki/index.md` 에 전체 프로젝트 요약표가 생긴다. 7일 넘게 커밋이 없는 프로젝트는
정체(💤)로, `STATUS.md` 에 `status: blocked` 라고 적어둔 프로젝트는 블로커(🚫)로 맨 위에
경고처럼 올라온다. 통틀어 남은 할 일은 `todos.md`, 모아둔 설계 결정은 `adr.md` 쪽에
정리된다. 프로젝트마다 상세 페이지도 따로 생겨서 최근 커밋 열 개, STATUS 메모, 미커밋
파일, 할 일, ADR 을 한 번에 본다.

만들면서 스스로 정해둔 규칙이 하나 있었다. **LLM-Wiki-Hub 는 어떤 프로젝트도 수정하지
않는다**는 것. `git log`, `git status`, 그리고 평문 파일 몇 개를 읽는 정도에서 그친다.
합치지도, 양방향으로 동기화하지도 않는다. 흩어진 저장소는 흩어진 채로 둔다. 그래서
부작용을 걱정할 일이 거의 없다. 몇 번을 돌려도 결과가 같고, 산출물인 `wiki/*.md` 는
언제 지워도 다시 만들어진다. 망가질 상태가 딱히 없으니 부담 없이 돌린다.

### 이득은 채워둔 만큼 나온다

솔직하게 말하면, 아무 git 저장소나 경로에 넣어도 동작은 한다. 마지막 커밋이 언제였고
정체됐는지, 미커밋이 몇 개인지 정도는 `git` 만 읽어도 나오니까. 그래서 LLM-Wiki 랑은
상관없이 그냥 "여러 저장소 활동 현황판"으로 써도 나쁘진 않다.

다만 이 도구가 좀 더 쓸모 있어지는 건 그 프로젝트가 LLM-Wiki 로 관리되고 있을 때인 것
같다. 상태·마일스톤·블로커 경고, 통합 할일, 통합 ADR 같은 건 전부 `STATUS.md`·할일
파일·`docs/adr/` 에서 긁어오는 거라, 거기가 채워져 있는 프로젝트일수록 현황판이
풍부해진다. 대충 채워둔 만큼 나온다고 보면 맞을 것 같다. 어떻게 보면 이 도구 자체가
LLM-Wiki 를 계속 쓰게 만드는 동기가 되기도 했다.

### 한계도 솔직히

좋은 점만 적으면 공평하지 않으니 한계도 적어둔다. 읽기 전용이라 대시보드에서 바로
뭘 고치거나 액션을 할 수는 없다. 상태를 바꾸려면 결국 그 프로젝트로 가서 `STATUS.md` 를
고치고 커밋해야 한다. 그리고 할일·ADR 수집이 규칙(파일 이름, 디렉토리) 기반이라,
자유 형식으로 적어둔 문서는 잘 못 잡는다. 실시간도 아니고 주기적으로 다시 긁는
방식이라, 방금 한 커밋을 보려면 한 번 재집계해야 한다. 혼자 로컬에서 쓰는 걸 전제로
만든 거라 팀 단위로 바로 쓰기엔 손볼 게 좀 있다.

## 프로젝트를 LLM-Wiki 로 만들어 두기

순서를 정리하면, 먼저 각 프로젝트를 LLM-Wiki 로 만들어 두는 게 먼저다. 나는 새
프로젝트를 시작할 때 대충 이렇게 한다.

루트에 `STATUS.md` 를 하나 둔다. 이건 LLM-Wiki-Hub 의 일부가 아니라 그 프로젝트 git 에
커밋하는 파일이다.

```markdown
# STATUS
status: active          # active / paused / blocked / done
milestone: 펌웨어 OTA 안정화
updated: 2026-06-27
blockers: 없음           # 막힌 게 있으면 적는다 → 대시보드 맨 위 🚫 로 뜸
```

할 일은 `TODO.md` 나 `PLAN.md`, `*-tasks.md` 같은 '할일 목적' 파일에 `- [ ]` 형식으로
적는다. 설계 문서나 README 에도 체크박스를 쓰긴 하는데 그런 것까지 긁으면 노이즈가
너무 심해서, 기본적으로는 할일 목적 파일만 보도록 좁혀뒀다. 설계 결정은 `docs/adr/` 에
한 장씩 쌓는다. 그리고 LLM 에게 줄 운영 규칙은 `CLAUDE.md` 에 적어둔다 — 위키를 어떻게
유지할지, 어떤 파일이 raw 이고 어떤 파일이 위키인지 같은 것들.

이 단계는 LLM-Wiki-Hub 와 무관하게 그냥 그 프로젝트 안에서 LLM 과 일하기 편해지는 작업이라,
LLM-Wiki-Hub 를 안 쓸 거여도 해두면 좋다고 생각한다.

## 위키들을 LLM-Wiki-Hub 로 묶기

그다음이 묶는 단계인데, 이건 정말 별거 없다. `projects.txt` 에 경로를 한 줄씩 추가하고
`aggregate.py` 를 돌리면 끝이다. 목록에서 뺀 프로젝트의 상세 페이지는 다음 집계 때
알아서 정리된다.

스크립트가 안에서 하는 일도 단순하다. 프로젝트별로 git 메타데이터(마지막 커밋, 브랜치,
미커밋 상태, 최근 커밋)를 읽고, `STATUS.md` 에서 상태와 메모를 파싱하고, 할일 파일과
ADR 디렉토리를 훑어서 markdown 으로 다시 써낸다. 어느 프로젝트도 건드리지 않고, 몇 번을
돌려도 같은 결과가 나오게(멱등) 만들어 뒀다.

## 브라우저로 보기 좋게 — 뷰어 붙이기

여기까지 만들고 한동안은 그냥 에디터에서 `wiki/index.md` 를 열어 봤다. 동작은 했는데,
프로젝트 상세 페이지로 링크를 타고 넘어가거나 표를 훑기엔 영 불편했다. 결국 원하는 건
별생각 없이 켜둘 수 있는 브라우저 탭 하나였다.

거창하게 갈 생각은 없었다. 빌드 단계나 프레임워크를 끼우고 싶지도 않았고. 그래서 고른
게 [docsify](https://docsify.js.org/) 였다. docsify 는 `index.html` 한 장이 CDN 에서
스크립트를 불러와서, markdown 을 **빌드 없이 브라우저에서 그때그때 렌더**해준다.
컴파일도 `node_modules` 도 없다. `aggregate.py` 가 이미 써둔 `*.md` 를 그대로 가져다
보여주기만 하면 되니, 내 경우엔 딱 맞았다.

그래서 `wiki/index.html` 을 docsify 셸로 직접 적어뒀다. 재밌는 건, `wiki/` 안에서 이
파일만은 집계가 다시 만들지 않는다는 점이다. 나머지 `*.md` 는 전부 파생물이라 갈아엎지만,
이 셸 한 장은 내가 관리하는 자산으로 남겨뒀다. 여기서 사이드바(`_sidebar.md`, 이건 또
집계가 만들어준다)랑 검색, 그리고 우상단 새로고침 버튼을 붙인다.

그런데 정적으로 파일만 띄우면 데이터가 금방 낡는다. 그래서 작은 서버를 하나 더 뒀다.
`scripts/serve.py` 인데, 하는 일은 두 가지다. `wiki/` 를 http 로 서빙하고, 시작할 때랑
이후 한 시간마다 `aggregate.py` 를 다시 돌린다. 여기에 `GET /refresh` 엔드포인트를
하나 달아서, 호출하면 즉시 재집계하고 다시 `/` 로 돌려보낸다 — 아까 그 우상단 ↻ 버튼이
누르는 게 이거다. 물론 이 서버도 단방향·멱등 원칙은 그대로라, 프로젝트는 건드리지 않고
그냥 같은 집계를 다시 돌릴 뿐이다.

마지막으로 이걸 매번 손으로 띄우긴 귀찮아서 pm2 로 감쌌다.

```
pm2 start ecosystem.config.cjs     # 백그라운드 구동
pm2 save && pm2 startup            # 부팅 시 자동 시작 (선택)
```

이러면 http://127.0.0.1:8787 에 항상 최신 현황이 떠 있고, 컴퓨터를 껐다 켜도 알아서
다시 올라온다. 포트나 재집계 주기는 `ecosystem.config.cjs` 의 `LLMWIKIHUB_PORT` /
`LLMWIKIHUB_REFRESH_SECONDS` 로 바꾼다. pm2 가 싫으면 그냥 `python scripts/serve.py` 로 띄워도
되고, 아예 브라우저 뷰가 필요 없으면 `scripts/` 안에 넣어둔 macOS launchd · Linux
systemd 예시로 주기 재집계만 돌린 다음 `*.md` 를 알아서 읽어도 된다.

## 돌아보며

만들고 나서 든 생각은, 결국 핵심은 도구가 아니라 그 밑의 발상 쪽이었다는 거다.
**LLM 이 읽을 평문을 코드 옆에 두고, 단일 출처를 유지하고, 정리는 자동으로 모은다.**
이 발상은 프로젝트 한 개에도 통했고, 그걸 한 층 위로 올리니 프로젝트 묶음에도 통했다.
같은 원리가 위로 한 번 더 재귀한 셈인데, 개인적으로는 그 지점이 제일 재밌었다.

아직 부족한 게 많다. 읽기 전용이라는 한계를 넘어서 대시보드에서 바로 상태를 바꾸게
한다든지, 실시간에 가깝게 만든다든지, 여러 머신에 흩어진 저장소까지 묶는다든지 —
하다 보면 욕심나는 것들이 생긴다. 그래도 일단 지금 수준에서도 매일 잘 쓰고 있으니,
비슷한 고민을 하는 사람이 있다면 한 번 가져다 써봐도 좋을 것 같다.

코드는 [LLM-Wiki-Hub 저장소](../README.md)에 있고, LLM-Wiki 원본 패턴은
[Karpathy 의 gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)
를 참고하면 된다.
